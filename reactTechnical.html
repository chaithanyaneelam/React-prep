01 · Foundations

// the vocabulary every round assumes you already have
01-jsx.jsx
BASIC

JSX (JavaScript XML)

Definition
A syntax extension that lets you write HTML-like markup directly inside JavaScript. It is not a template language — it compiles (via Babel) into nested React.createElement() calls that return plain JS objects.
Why we use it
Before JSX, UI trees were built by hand-nesting createElement calls, which was unreadable at scale. JSX keeps markup and the logic that drives it in one file, so structure and behavior stay visually and mentally coupled.
Example
const Greeting = ({ name }) => {
  return (
    <h1 className="title">
      Hello, {name}! {/* JS expression inline */}
    </h1>
  );
};
02-components.jsx
BASIC

Components (Functional vs Class)

Definition
A component is a self-contained, reusable function or class that returns JSX. Functional components are plain JS functions using Hooks for state/lifecycle; class components extend React.Component and use this.state and lifecycle methods.
Why we use it
Components let you break a UI into independent, testable pieces and compose them like building blocks, instead of one monolithic template. Functional components + Hooks (post-2019) removed the need for this binding and made logic reuse far simpler than class-based mixins/HOC chains.
Example
// Functional (modern standard)
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}

// Class (legacy / still asked in interviews)
class Counter extends React.Component {
  state = { count: 0 };
  render() { return <button onClick={() => this.setState({count: this.state.count+1})}>{this.state.count}</button>; }
}
03-props.jsx
BASIC

Props

Definition
Read-only inputs passed from a parent component to a child, similar to function arguments. Props flow in one direction only — top to bottom (unidirectional data flow).
Why we use it
Props let a component stay generic and configurable instead of hardcoding data, so the same Button or Card component can render differently in every context it's used. Immutability of props keeps data flow predictable and easy to trace during debugging.
Example
function UserCard({ name, role }) {
  return <p>{name} — {role}</p>;
}
// Parent passes data down:
<UserCard name="Chaithanya" role="Founder" />
04-usestate.jsx
BASIC

State & useState

Definition
State is data local to a component that can change over time. useState is the Hook that adds state to a functional component, returning a value and a setter function.
Why we use it
Regular JS variables don't survive re-renders and mutating them doesn't trigger a re-render at all. useState gives React a value it tracks internally — calling the setter both persists the new value across renders and schedules a re-render so the UI stays in sync with data.
Example
const [quantity, setQuantity] = useState(1);
// Functional update avoids stale-closure bugs
setQuantity(prev => prev + 1);
05-events.jsx
BASIC

Event Handling

Definition
React wraps native DOM events in a cross-browser wrapper called a SyntheticEvent, and events are attached using camelCase props like onClick, onChange.
Why we use it
SyntheticEvents normalize inconsistent browser event APIs into one predictable interface, and React batches/pools them for performance. It also lets React attach a single listener at the root and delegate internally, rather than one raw listener per DOM node.
Example
<input
  onChange={(e) => setEmail(e.target.value)}
  onKeyDown={(e) => { if (e.key === "Enter") submit(); }}
/>
06-conditional.jsx
BASIC

Conditional Rendering

Definition
Showing different UI based on state or props, using plain JavaScript operators (ternary, &&, early return) instead of a special template directive.
Why we use it
Because JSX is "just JavaScript", React deliberately avoids inventing template syntax like v-if. This keeps the mental model small — anything you can do in JS, you can do in JSX — but interviewers probe this because && with a falsy 0 is a classic bug (renders a stray "0").
Example
{isLoggedIn ? <Dashboard /> : <LoginForm />}
{cartItems.length > 0 && <CartBadge count={cartItems.length} />}
07-lists-keys.jsx
BASIC

Lists and Keys

Definition
Rendering an array of data with .map(), where each rendered element needs a stable, unique key prop.
Why we use it
Keys give React's diffing algorithm an identity for each item across renders, so it can detect "this item moved" instead of "everything after index 2 changed" — avoiding unnecessary unmount/remount cycles and preserving component state (like input focus) correctly. Using array index as a key breaks this when the list is reordered or filtered.
Example
{todos.map(todo => (
  <TodoItem key={todo.id} text={todo.text} />
))}
08-forms.jsx
BASIC

Forms & Controlled Components

Definition
A controlled component's form value is driven entirely by React state — the input's value comes from state, and every keystroke updates that state via onChange.
Why we use it
Without control, the DOM holds its own source of truth for the input value, which drifts out of sync with your component's state — making validation, reset, and conditional logic unreliable. Making React the single source of truth lets you validate, format, or block input on every keystroke.
Example
const [email, setEmail] = useState("");
<input value={email} onChange={(e) => setEmail(e.target.value)} />

02 · Intermediate

// where "I've used React" turns into "I understand React"
09-useeffect.jsx
INTER

useEffect

Definition
A Hook that runs "side effects" — code that reaches outside the component (fetching data, subscriptions, timers, manual DOM work) — after React commits changes to the DOM.
Why we use it
Rendering must stay a pure calculation of UI from state/props; side effects don't belong in the render body because they can run multiple times or in the wrong order. useEffect gives side effects a dedicated, controllable lifecycle stage, with a dependency array to decide exactly when they re-run, and a cleanup function to prevent leaks.
Example
useEffect(() => {
  const id = setInterval(tick, 1000);
  return () => clearInterval(id); // cleanup on unmount
}, []); // empty array = run once on mount
10-useref.jsx
INTER

useRef

Definition
A Hook returning a mutable {`{ current }`} object that persists for the component's lifetime without causing a re-render when mutated. Commonly used to hold direct DOM node references.
Why we use it
Sometimes you need to keep a value across renders (a timer ID, previous prop value, DOM node) but changing it shouldn't trigger a re-render — useState would over-trigger renders here. useRef is the escape hatch for imperative access, like focusing an input or measuring an element's size.
Example
const inputRef = useRef(null);
useEffect(() => { inputRef.current.focus(); }, []);
<input ref={inputRef} />
11-usecontext.jsx
INTER

useContext

Definition
A Hook that reads a value from the nearest matching <Context.Provider> above it in the tree, without it being passed explicitly as a prop.
Why we use it
Passing data through 5-6 nested components that don't need it themselves — just to reach a deeply nested child — is "prop drilling" and makes refactors painful. Context lets any descendant subscribe directly to shared data like theme, auth user, or locale.
Example
const ThemeContext = createContext("light");
// Provider (ancestor)
<ThemeContext.Provider value="dark"><App /></ThemeContext.Provider>
// Consumer (any descendant)
const theme = useContext(ThemeContext);
12-custom-hooks.jsx
INTER

Custom Hooks

Definition
A JS function whose name starts with use and which calls other Hooks internally, letting you extract and reuse stateful logic across components.
Why we use it
Before Hooks, sharing stateful logic (e.g. "track window width") meant Higher-Order Components or render props, both of which added extra wrapper layers to the tree ("wrapper hell"). Custom Hooks share logic without adding any component to the tree at all.
Example
function useWindowWidth() {
  const [width, setWidth] = useState(window.innerWidth);
  useEffect(() => {
    const onResize = () => setWidth(window.innerWidth);
    window.addEventListener("resize", onResize);
    return () => window.removeEventListener("resize", onResize);
  }, []);
  return width;
}
13-lifting-state.jsx
INTER

Lifting State Up

Definition
Moving shared state from a child component to their closest common parent, which then passes it back down as props.
Why we use it
Two sibling components can't talk to each other directly since data flow is one-directional. Lifting state to a shared ancestor gives both siblings a single source of truth, keeping them in sync without ad-hoc communication channels.
Example
function Parent() {
  const [temp, setTemp] = useState(24);
  return (<>
    <CelsiusInput value={temp} onChange={setTemp} />
    <FahrenheitDisplay celsius={temp} />
  </>);
}
14-composition.jsx
INTER

Component Composition (children prop)

Definition
Building complex UI by nesting components inside one another via the special children prop, rather than through configuration props alone.
Why we use it
React deliberately favors composition over inheritance — instead of subclassing a Panel component to customize it, you wrap it around arbitrary content. This keeps components flexible ("slots") and avoids deep, fragile inheritance hierarchies.
Example
function Card({ children }) {
  return <div className="card">{children}</div>;
}
<Card><h2>Plan</h2><p>Pro tier</p></Card>
15-error-boundaries.jsx
INTER

Error Boundaries

Definition
A class component implementing static getDerivedStateFromError() and/or componentDidCatch() that catches JS errors thrown anywhere in its child tree during render and shows a fallback UI instead of crashing the whole app. There is no Hook equivalent yet.
Why we use it
By default, one uncaught render error unmounts the entire React tree, giving users a blank white screen. Error boundaries contain the blast radius to just the failing subtree, so a broken widget doesn't take down the whole page.
Example
class ErrorBoundary extends React.Component {
  state = { hasError: false };
  static getDerivedStateFromError() { return { hasError: true }; }
  render() {
    return this.state.hasError ? <Fallback /> : this.props.children;
  }
}

16-usereducer.jsx
ADV

useReducer

Definition
A Hook for managing state via a reducer function (state, action) => newState — the same pattern Redux popularized — dispatching actions instead of calling setters directly.
Why we use it
When state updates depend on multiple sub-values that change together, or the next state depends on the previous one in complex ways, scattered useState calls get hard to reason about and prone to inconsistent intermediate states. Centralizing "what changed" (action) and "how it changes" (reducer) in one place makes transitions predictable and testable in isolation.
Example
function reducer(state, action) {
  switch (action.type) {
    case "increment": return { count: state.count + 1 };
    default: throw new Error();
  }
}
const [state, dispatch] = useReducer(reducer, { count: 0 });
dispatch({ type: "increment" });
17-usememo.jsx
ADV

useMemo

Definition
A Hook that caches (memoizes) the result of an expensive calculation, recomputing it only when its dependencies change.
Why we use it
Every re-render re-runs the whole function body by default, including expensive calculations (sorting thousands of rows, heavy math) that didn't need to change. useMemo skips that recalculation when its inputs are unchanged, trading a bit of memory for saved CPU work.
Example
const sortedList = useMemo(
  () => expensiveSort(items),
  [items]
);
18-usecallback.jsx
ADV

useCallback

Definition
A Hook that memoizes a function reference itself, returning the same function instance across renders unless its dependencies change.
Why we use it
In JS, a function defined inside a component body is a brand-new object on every render — this breaks reference-equality checks like React.memo on a child, causing it to re-render even though "nothing really changed". useCallback keeps the reference stable so memoized children can actually skip re-rendering.
Example
const handleAdd = useCallback(
  (item) => setItems(prev => [...prev, item]),
  [] // stable reference, passed to a memoized child
);
19-react-memo.jsx
ADV

React.memo

Definition
A higher-order component that wraps a component and skips its re-render if its props are shallowly equal to the previous render's props.
Why we use it
By default, a parent re-rendering re-renders every child regardless of whether that child's own props actually changed. React.memo adds a props-comparison gate in front of a component, useful for expensive "leaf" components (charts, large lists) — but pointless (even counterproductive) on cheap components, since the comparison itself has a cost.
Example
const ProductRow = React.memo(function ProductRow({ product }) {
  return <tr><td>{product.name}</td></tr>;
});
20-hoc.jsx
ADV

Higher-Order Components (HOC)

Definition
A function that takes a component and returns a new, enhanced component — a pattern borrowed from functional programming's higher-order functions.
Why we use it
HOCs let you inject shared behavior — authentication checks, logging, data-fetching — into many unrelated components without duplicating that logic in each one. Custom Hooks now cover most of these use cases more simply, so HOCs mainly appear in legacy codebases and libraries like react-redux's connect().
Example
function withAuth(Component) {
  return function Wrapped(props) {
    const user = useAuth();
    return user ? <Component {...props} /> : <Redirect to="/login" />;
  };
}
const ProtectedDashboard = withAuth(Dashboard);
21-render-props.jsx
ADV

Render Props

Definition
A pattern where a component takes a function as a prop (often called render or passed as children) and calls it to determine what to render, sharing its internal state with that function.
Why we use it
Like HOCs, render props solved cross-component logic sharing before Hooks existed — a <Mouse> component tracking cursor position could hand that data to any UI via a function, without dictating what gets rendered. Mostly superseded by custom Hooks today, but still shows up in library APIs (e.g. React Router's older versions, Downshift).
Example
<DataFetcher url="/api/user">
  {(data, loading) => loading ? <Spinner /> : <Profile data={data} />}
</DataFetcher>
22-lazy-suspense.jsx
ADV

Code Splitting: React.lazy + Suspense

Definition
React.lazy() dynamically imports a component so it's bundled into a separate chunk; <Suspense> shows a fallback UI while that chunk is still loading.
Why we use it
Shipping one giant JS bundle means users download code for routes/features they may never open, slowing the initial load. Code splitting loads only what's needed for the current view, deferring the rest until it's actually requested — directly improving Time to Interactive.
Example
const Settings = React.lazy(() => import("./Settings"));
<Suspense fallback={<Spinner />}>
  <Settings />
</Suspense>
23-portals.jsx
ADV

Portals

Definition
ReactDOM.createPortal(child, domNode) renders a component's output into a DOM node outside its parent's DOM hierarchy, while keeping it inside the same React component tree (event bubbling and context still work normally).
Why we use it
Elements like modals, tooltips, and dropdowns need to visually escape a parent's overflow:hidden or low z-index stacking context, but still behave like a child in React's data flow. Portals solve exactly that split between "visual DOM position" and "logical component position".
Example
function Modal({ children }) {
  return ReactDOM.createPortal(
    <div className="modal">{children}</div>,
    document.getElementById("modal-root")
  );
}
24-fiber.jsx
ADV

Virtual DOM, Reconciliation & Fiber

Definition
The Virtual DOM is a lightweight JS object tree mirroring the real DOM. Reconciliation is the diffing algorithm that compares the new Virtual DOM tree with the previous one to compute the minimal set of real DOM changes. Fiber (React 16+) is the reconciler's internal data structure and engine — it breaks rendering work into interruptible units.
Why we use it
Direct DOM manipulation is slow because layout/paint gets recalculated on every mutation. Diffing in-memory JS objects first, then batching real DOM writes, is far cheaper. Fiber specifically lets React pause, abort, or prioritize rendering work (e.g. yield to an urgent keystroke over a background list update) — impossible with the old synchronous stack-based reconciler.
Example
// Conceptual: React compares fiber trees by type + key
// same type -> update in place; different type -> unmount + remount
<div> → <div>   // diffed, attributes patched
<div> → <span>  // old subtree destroyed, new one built

04 · Crucial Comparisons

// the questions every interviewer eventually asks
VS

useState vs useEffect

Aspect useState useEffect
Purpose Declares and updates a piece of local component state Runs side effects in response to render / dependency changes
Syntax const [v, setV] = useState(init) useEffect(() => {...}, [deps])
Runs Setter call triggers a re-render immediately (state updates are batched) After the DOM has committed / painted, not during render
Use case Form inputs, toggles, counters — data the UI directly reflects Data fetching, subscriptions, timers, manually syncing with a non-React system
Performance note Excess state causes excess re-renders of that component Missing/incorrect dependency arrays cause stale data bugs or infinite loops
VS

useState vs useReducer

Aspect useState useReducer
Purpose Manage a single, simple, independent value Manage complex state with multiple sub-values or interdependent transitions
Syntax const [v, setV] = useState(init) const [state, dispatch] = useReducer(reducer, init)
Update model Direct setter call with a new value or updater function Dispatch a described action; reducer decides the next state
Use case Toggle, text input, counter Multi-step forms, shopping cart, undo/redo, state machines
Performance note Simpler, less boilerplate, fine-grained updates Centralizes logic outside the component — easier to test and predict for complex flows, at the cost of more setup
VS

useMemo vs useCallback

Aspect useMemo useCallback
Purpose Memoizes a computed value Memoizes a function reference
Syntax useMemo(() => computeX(a,b), [a,b]) useCallback(() => doX(a,b), [a,b])
Returns The result of calling the function The function itself, uncalled
Use case Expensive calculations (filtering/sorting large arrays) Stable callback identity for props passed to React.memo children or effect dependencies
Performance note Relation: useCallback(fn, deps) is equivalent to useMemo(() => fn, deps). Both add memory overhead — don't wrap everything by default Same caveat: only pays off when the reference stability actually prevents downstream work
VS

useEffect vs useLayoutEffect

Aspect useEffect useLayoutEffect
Timing Asynchronous — runs after the browser has painted the screen Synchronous — runs after DOM mutations but before the browser paints
Use case Data fetching, logging, subscriptions — anything not visually blocking Reading/adjusting DOM layout (measuring an element, preventing a visual flicker)
Performance note Doesn't block painting, so the UI feels more responsive Blocks the paint until it finishes — overusing it can noticeably slow perceived performance
VS

Controlled vs Uncontrolled Components

Aspect Controlled Uncontrolled
Source of truth React state (value + onChange) The DOM itself, accessed via ref when needed
Syntax <input value={v} onChange={...}/> <input ref={r} defaultValue="x"/>
Use case Real-time validation, conditional disabling, formatting as you type Simple forms, file inputs (always uncontrolled), quick integrations with non-React code
Performance note Every keystroke re-renders the owning component No re-renders on typing — cheaper for very large, simple forms
VS

Context API vs Redux (external state library)

Aspect Context API Redux
Purpose Avoid prop drilling for relatively static/simple shared data Predictable, centralized state management with strict update rules for complex apps
Syntax createContext + Provider + useContext createSlice/reducers + store + useSelector/useDispatch
Tooling Built into React, no dev tools Redux DevTools — time-travel debugging, action logs
Performance note Any Provider value change re-renders all consumers, even ones using only part of the value — needs manual splitting Selectors let components subscribe to only the state slice they need, minimizing re-renders even at large scale
VS

React.memo vs useMemo

Aspect React.memo useMemo
What it wraps An entire component A single computed value inside a component
Syntax React.memo(MyComponent) useMemo(() => value, [deps])
Use case Prevent a child component from re-rendering when its props haven't changed Prevent an expensive calculation from re-running inside a component that's already re-rendering
Performance note Comparison happens once per render at the component boundary Comparison happens once per render at the value level; both are wasted effort if props/deps change every render anyway
VS

Class Components vs Functional Components

Aspect Class Components Functional Components
State this.state + this.setState() useState Hook
Lifecycle componentDidMount, componentDidUpdate, componentWillUnmount useEffect covers all three via dependency arrays
Logic reuse HOCs / render props (extra wrapper components) Custom Hooks (no extra components)
Performance note Slightly more boilerplate; this binding is a common bug source Generally smaller bundle output and easier to optimize with Hooks like useMemo/useCallback
